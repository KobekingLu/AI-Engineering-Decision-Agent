# NotebookLM 提問模板包 v2｜AI Engineering Decision Agent

## 這份文件的用途
這份文件提供更適合你目前 NotebookLM 專案的提問模板。
目前專案已上傳：
- 01_project_brief_notebooklm.md
- 02_submission_form_answers_notebooklm.md
- 03_slide_source_notebooklm.md
- 04_case_examples_notebooklm.md

建議優先使用的 NotebookLM 工作室功能如下：
1. 簡報（Slide Deck）
2. 報告（Briefing / Report）
3. 心智圖（Mind Map）
4. 影片 / Video Overview（作為 demo 腳本輔助）
5. 學習卡 / 測驗（僅作為評審 Q&A 自我演練，不是主輸出）

---

## 優先順序建議

### 第一優先｜簡報
用途：
- 直接生成競賽簡報初稿
- 幫你快速檢查主線是否清楚
- 適合先拿來做第一輪版型與頁面分配

### 第二優先｜報告
用途：
- 幫你生成較完整的提案摘要
- 可拿來當報名表長版草稿、主管版摘要、OneDrive 說明文件基底

### 第三優先｜心智圖
用途：
- 檢查作品主線是否過散
- 幫你確認評審看到的核心節點是否集中在：
  - 痛點
  - Agent 定位
  - decision workflow
  - triage / verification / closure
  - 目前成果
  - 未來方向

### 第四優先｜影片 / Video Overview
用途：
- 當作 demo 影片腳本靈感
- 幫你檢查口語敘事順不順
- 但不建議直接原封不動拿去投稿，仍需人工調整

### 第五優先｜學習卡 / 測驗
用途：
- 自我練習評審問答
- 幫助你找出目前專案哪幾個地方最容易被問倒
- 不作為正式投稿素材

---

## 提問模板 1｜請幫我做競賽簡報初稿
請根據目前專案來源，幫我產出一份 10~12 頁的競賽簡報初稿。

要求如下：
- 使用繁體中文
- 風格像工程競賽提案，不要像行銷文案
- 主線聚焦在「把 scattered evidence 收斂成 decision-ready view 的 AI Agent」
- 要明確說出這不是 chatbot，而是具備 decision workflow 的 AI Agent
- 結構請排成：痛點 → 核心定位 → 為何是 Agent → 技術架構 → demo output → 代表案例 → 差異化 → 目前成果 → 未來方向
- 每頁請提供：頁標題、3~5 個 bullet、建議口播重點
- 請保守表述成果，不要虛構未驗證的量化數字
- 請優先使用 triage / verification / closure 三個 decision stage 來組織 demo 案例

若內容過度像內部開發文件，請幫我改寫成更適合競賽評審閱讀的版本。

---

## 提問模板 2｜請幫我做 OneDrive 用提案報告
請根據目前專案來源，幫我整理一份適合放在 OneDrive 資料夾中的提案報告文字版。

要求如下：
- 用繁體中文
- 語氣正式、可信、工程導向
- 內容包含：
  - 專案摘要
  - 問題痛點
  - 解決方案
  - 為什麼這是 Agent
  - 目前完成度
  - 代表案例
  - 可保守主張的成果
  - 未來方向
- 不要浮誇，也不要把 prototype 說成 production-ready
- 長度控制在可整理成 1~2 頁說明文件的程度

---

## 提問模板 3｜請幫我直接填競賽報名表
請根據目前專案來源，直接幫我填寫競賽報名表需要的內容。

請產出以下欄位：
- 作品名稱（Project Title）
- 最相關領域（Relevant Field）
- 作品連結（Project Link）描述文字
- Demo / 影片連結描述文字
- 應用場景（Application Scenario）
- 實施方法與技術架構（Implementation Method & Technical Architecture）
- 量化成效與實施成果（Quantitative Benefits & Implementation Results）
- 未來展望（Future Outlook）
- 其他補充資料（Additional Information）

要求如下：
- 每一欄請先給正式版，再給精簡版
- 語氣正式、可信、工程導向
- 不要浮誇，也不要把 prototype 說成 production-ready
- 要明確凸顯 autonomous decision-making 與 environment interaction
- 若量化成效無法嚴格證明，請用保守可信的說法處理

---

## 提問模板 4｜請幫我整理心智圖主線
請根據目前專案來源，幫我整理一份適合做成心智圖的主線結構。

要求如下：
- 第一層節點請不要超過 6 個
- 請優先集中在：
  - 業務痛點
  - 核心定位
  - Agent workflow
  - 代表案例
  - 目前成果
  - 未來方向
- 每個第一層節點下，再展開 3~5 個第二層節點
- 要讓評審看完心智圖後，能快速理解這不是 generic chatbot，而是工程決策輔助 Agent
- 若某些節點太偏內部開發術語，請改寫成較好懂的版本

---

## 提問模板 5｜請幫我做 3 分鐘 demo 口播稿
請根據目前專案來源，幫我寫一份 3 分鐘 demo 口播稿。

要求如下：
- 用繁體中文
- 口吻自然但專業
- 開頭先講痛點，再講解法，再講為什麼是 Agent
- 中間請用 triage / verification / closure 的案例邏輯帶出作品能力
- 最後收斂到目前成果與未來方向
- 不要太像背稿，要像比賽 demo 口頭說明
- 請避免過度技術細節堆疊，優先讓評審聽懂主價值

---

## 提問模板 6｜請幫我做 5 分鐘 demo 影片腳本
請根據目前專案來源，幫我寫一份 5 分鐘 demo 影片腳本。

要求如下：
- 用時間軸方式呈現，例如 0:00–0:30、0:30–1:30
- 說明這個作品要解的問題、系統怎麼運作、輸出長什麼樣子、案例如何證明能力
- 至少納入一個 triage case、一個 verification case、一個 closure case
- 語氣要像工程競賽 demo，不要太像廣告
- 腳本請保留「可搭配畫面」提示，例如：
  - 此段適合放系統流程圖
  - 此段適合放 HTML output 截圖
  - 此段適合放三個案例對照圖

---

## 提問模板 7｜請幫我準備評審 Q&A
請根據目前專案來源，幫我準備 10 題評審可能會問的問題，以及每題的建議回答。

請至少包含以下主題：
- 為什麼這是 Agent，不是 chatbot？
- 目前與一般 AI 工具的差異是什麼？
- 目前是否已有量化成效？
- 沒有最終答案時，Agent 為何仍有價值？
- 如何避免錯誤知識被累積進去？
- 未來如何串接更多資料來源？
- 為什麼這個方向值得持續投資？

要求如下：
- 回答要保守、可信
- 不要虛構 production 成果
- 要能凸顯 engineering decision assistance 的價值
- 若某題目前證據不足，請明確用「目前可保守表述為…」來回答

---

## 提問模板 8｜請幫我濃縮成主管 / 評審 1 頁版
請根據目前專案來源，幫我濃縮成一頁摘要，讓主管或評審可以在 1 分鐘內看懂。

要求如下：
- 只保留最有說服力的內容
- 包含：痛點、解法、為什麼是 Agent、目前完成度、代表案例、未來方向
- 風格偏高層 summary，但不要失真
- 不要變成空泛口號

---

## 提問模板 9｜若生成結果偏掉，請這樣修正
若你生成的內容出現以下問題，請重寫：
- 把作品寫成 generic chatbot
- 過度強調「AI 已自動找出答案」
- 虛構量化 ROI
- 沒有凸顯 triage / verification / closure 的 decision stage coverage
- 沒有明確說明 lifecycle signal 與 next-step guidance
- 太像內部開發文件，不像競賽提案
- 案例講太多 bug 細節，卻沒有說明案例證明了什麼能力

重寫時請優先做到：
- 強化作品主線
- 強化 Agent 定位
- 強化 demo 可講性
- 保持保守可信
